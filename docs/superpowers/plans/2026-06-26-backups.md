# Backups (Scheduled + On-demand) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add on-demand and scheduled backups of databases and website files to Laranode. Backups run as `OperationJob` subclasses (live progress + audit), store to local disk or any S3-compatible bucket, prune on `retention_count`, and support restore to a new named target only.

**Architecture:** Controller → FormRequest → Service (creates rows + dispatches job) → `BackupJob`/`RestoreJob` extends `OperationJob`. Storage uses `Storage::disk()` — local and S3 behind the same interface. Scheduler: `RunScheduledBackupsJob` (`ShouldBeUnique`) dispatched `everyMinute()` via the existing `withSchedule` in `bootstrap/app.php`; it evaluates `CronExpression::isDue()` per `ScheduledBackup` row.

## Global Constraints

- **Depends on #1 `platform-async-progress` (shipped).** `OperationJob`, `Operation` model, `OperationUpdated` event, `bootstrap/app.php` `withSchedule` hook, `useOperation` hook, `<OperationProgress>` component — all present and must not be modified.
- **`BackupJob` and `RestoreJob` extend `App\Jobs\OperationJob`** exactly as `GenerateSslOperationJob` does.
- **`scopeMine(Builder)`** on `Backup` and `ScheduledBackup` mirrors `Database::scopeMine()` (`app/Models/Database.php:49`) exactly.
- **S3 credentials on `ScheduledBackup`** use `'encrypted'` cast — same pattern as `Database::$db_password`. Never logged, never in broadcast payload, never in job constructor payload.
- **DB password to dump script via `--defaults-extra-file`** (temp `.cnf`, mode 0600, deleted in `finally`). Never in argv.
- **DB restore routes through `CreateDatabaseService`** — no bare `CREATE DATABASE`; the restored DB gets a panel row + per-site MySQL user.
- **`new_target` identifier validation** — regex `/^[a-zA-Z0-9_]{1,64}$/` enforced in `RestoreBackupRequest` AND re-checked in `RestoreJob::run()`.
- **S3 dependency** — `composer require league/flysystem-aws-s3-v3` is Task 0 (before any S3 code).
- **`backups` disk** defined in `config/filesystems.php` with root `env('BACKUP_LOCAL_ROOT', '/home')`. All backup jobs use `Storage::disk('backups')` for local storage.
- **`disk_name` persisted on `Backup` row** at creation — `download`, `delete`, and `retain` all read it from there.
- **`RunScheduledBackupsJob` implements `ShouldBeUnique`** to prevent `everyMinute()` double-fires. Additional guard: skip entries whose `last_run_at` is within 50 seconds.
- **`ScheduledBackupPolicy`** must gate `destroySchedule` — without it any authenticated user can delete any schedule.
- **Privileged scripts** go in `laranode-scripts/bin/`; sudoers entries go in `laranode-scripts/etc/sudoers.d/laranode-panel` (an explicit list, replacing the existing wildcard — do not add a second conflicting drop-in).
- **File backup scope is `websiteRoot`** (full site directory).
- **Download** — local: stream via `Storage::disk()->readStream()`; S3: presigned URL redirect.
- **Tests run with `QUEUE_CONNECTION=sync`** (already in `phpunit.xml`). Use `Event::fake()`, `Process::fake()`, `Storage::fake('backups')`. System-touching tests gated behind `LARANODE_SYSTEM_TESTS=1`.
- **Branch:** `feature/backups` (off `main`). Each task commits here.
- **Run the suite in the `local-dev` container**: `docker exec laranode-lab bash -lc 'cd /home/laranode_ln/panel && php artisan test'`. On Windows, use PowerShell for `make`/`docker compose`.
- **Pint before every PHP commit:** `./vendor/bin/pint --test` must pass cleanly.

**Key design points:**

**Acceptance (checked at Task 10 system tests):** Scripts are executable; `visudo -c -f /etc/sudoers.d/laranode-panel` exits 0; `sudo laranode-db-backup.sh ...` runs without password prompt as `www-data`.

**Key design points:**

`laranode-db-backup.sh <engine> <dbName> <dbUser> <cnfFile> <outFile>`:
```bash
set -euo pipefail
ENGINE="$1"; DB_NAME="$2"; DB_USER="$3"; CNF_FILE="$4"; OUT_FILE="$5"
case "$ENGINE" in
  mysql)
    mysqldump --defaults-extra-file="$CNF_FILE" --user="$DB_USER" \
      --single-transaction --quick --lock-tables=false "$DB_NAME" | gzip > "$OUT_FILE"
    ;;
  *) echo "Unsupported engine: $ENGINE" >&2; exit 1 ;;
esac
```

`laranode-restore-db.sh <cnfFile> <dumpFile> <dbName>`:
```bash
set -euo pipefail
zcat "$2" | mysql --defaults-extra-file="$1" "$3"
```

`laranode-backup-files.sh <siteRoot> <outFile> <sysUser>`: `tar czf $outFile -C $siteRoot .`; `chown www-data:www-data $outFile`.

`laranode-restore-files.sh <tarFile> <destDir> <sysUser>`: `mkdir -p $destDir && tar xzf $tarFile -C $destDir`; `chown -R $sysUser:$sysUser $destDir`.

Sudoers (`laranode-scripts/etc/sudoers.d/laranode-panel`, mode 0440): replace the monolithic wildcard with one explicit `NOPASSWD` line per script (all existing scripts + the 4 new ones). The installer must copy this file to `/etc/sudoers.d/laranode-panel` (and no longer append to `/etc/sudoers`).

**Acceptance:** `php artisan test --filter=BackupJobTest` passes (4 tests): `BackupJob` (db type) marks backup `completed` and operation `succeeded`; `BackupJob` (files type) marks backup `completed`; on dump failure, backup stays `pending` and operation is `failed`; `BackupService::handle()` creates `Backup` row, `Operation` row, returns `Operation`, and backup status ends `completed` (sync queue).

**Key design points:**
- `BackupJob(Operation $operation, Backup $backup)` — no S3 config in constructor. Disk resolved from `$this->backup->disk_name`.
- `run(callable $emit)`: dump/tar to temp file → `UploadToStorageAction` → update `Backup` → return 0. Temp file removed in `finally`.
- Storage path: `"{userId}/{type}/{target}/{Y-m-d-His}.{ext}"` — this is relative to the disk root (`/home`), so the full path resolves to e.g. `/home/1/db/mydb/2026-06-26-020000.sql.gz`. This stays within the `backups` disk.
- `BackupService::handle()`: for local storage, `disk_name = 'backups'`; for on-demand S3, registers a runtime disk keyed `'backups_s3'` from the request's S3 params (not persisted), sets `disk_name = 'backups_s3'` on the `Backup` row.

### Task 6: `RestoreJob` + `RestoreService` (TDD)

**Files:**
- Create: `app/Jobs/RestoreJob.php`
- Create: `app/Services/Backups/RestoreService.php` (+ `RestoreException`)
- Create: `tests/Feature/Backups/RestoreJobTest.php`

**Acceptance:** `php artisan test --filter=RestoreJobTest` passes (4 tests): `RestoreJob` (db type) calls `CreateDatabaseService` and marks operation `succeeded`; `RestoreJob` throws `InvalidArgumentException` when `new_target === source`, marks operation `failed`; `RestoreJob` throws `InvalidArgumentException` when `new_target` fails identifier regex, marks operation `failed`; `RestoreService::handle()` creates `Operation` row and returns it.

**Key design points:**
- `RestoreJob(Operation $operation, Backup $backup, string $newTarget)`.
- `run(callable $emit)`:
  1. Re-validate `$newTarget`: `preg_match('/^[a-zA-Z0-9_]{1,64}$/', $newTarget)` and `$newTarget !== $backup->target`. Throw `InvalidArgumentException` if either fails (OperationJob base catches this, marks failed, rethrows).
  2. Download backup from `Storage::disk($backup->disk_name)->readStream($backup->path)` to a temp file.
  3. For `db`: call `CreateDatabaseService` with `['name' => $newTarget, 'db_user' => $newTarget . '_u', 'db_pass' => Str::random(16), 'charset' => 'utf8mb4', 'collation' => 'utf8mb4_unicode_ci']` and `$backup->user`. Then pipe dump via `sudo laranode-restore-db.sh $cnfFile $tempFile $newTarget` (cnf written from the new DB's password).
  4. For `files`: call `sudo laranode-restore-files.sh $tempFile $destDir $sysUser`.
  5. Temp file removed in `finally`.
